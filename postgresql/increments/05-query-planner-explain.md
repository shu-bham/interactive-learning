# 🎯 Increment 05: Query Planner & EXPLAIN ANALYZE

**Duration**: 60 minutes  
**Difficulty**: ⭐⭐⭐⭐ Deep Dive

## 📋 Quick Summary

PostgreSQL has one of the most advanced Cost-Based Optimizers (CBO). Unlike MySQL, which often forces you to use `FORCE INDEX`, PostgreSQL's planner is usually correct if your statistics are fresh.

**Key Concepts**:
- **Statistics**: PostgreSQL collects detailed histograms of your data.
- **Scan Types**: Seq Scan (Scan all), Index Scan (One by one), Bitmap Scan (The "Secret Sauce").
- **Join Strategies**: Nested Loop, Merge Join, Hash Join (MySQL only recently got Hash Joins).
- **EXPLAIN ANALYZE**: The "Source of Truth" for query performance.

---

## 🎓 Theory (25 minutes)

### 1. The Cost Model

PostgreSQL assigns a "cost" to every operation.
- `seq_page_cost`: 1.0 (Reading 1 page sequentially)
- `random_page_cost`: 4.0 (Reading 1 page randomly - SSDs should set this to 1.1)
- `cpu_tuple_cost`: 0.01 (Processing 1 row)

The planner calculates the total cost for multiple paths and picks the cheapest one.

### 2. Scan Types

| Type | When it's used | Notes |
|------|---------------|-------|
| **Seq Scan** | Large table or no index | Traditional table scan. |
| **Index Scan** | Small result set via index | Finds PK, then fetches from Heap. |
| **Index Only Scan** | Covering Index | Never touches the Heap! Very fast. |
| **Bitmap Index Scan** | Middle ground | Finds all relevant pages via index, sorts them, then reads sequentially. |

### 3. Join Strategies (Senior MUST-KNOW)

| Strategy | Behavior | Best for... |
|----------|----------|-------------|
| **Nested Loop** | For each row in A, look up in B | Small inner table + index on outer. |
| **Hash Join** | Build hash table for A, probe with B | Large tables where nesting is too slow. |
| **Merge Join** | Sort A, Sort B, then zip them | Already sorted data (indexes) or very large joins. |

## 🧪 Hands-On Exercises (25 minutes)

### Exercise 1: EXPLAIN vs EXPLAIN ANALYZE

`EXPLAIN` shows the **plan**. `EXPLAIN ANALYZE` actually **executes** it and shows real timings.

```sql
-- 1. Just the plan (Fast, no execution)
EXPLAIN SELECT * FROM users WHERE user_id = 1;

-- 2. Execution stats (Slower, shows reality)
EXPLAIN ANALYZE SELECT * FROM users WHERE user_id = 1;

-- 3. Level Up: Add Buffers (Shows memory usage)
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM users WHERE user_id = 1;
```

### Exercise 2: The Bitmap Scan Magic

If you search for something that returns ~10% of the table, PostgreSQL often picks a **Bitmap Scan**.

```sql
-- Let's force it on our sample data (even if it's small)
SET enable_seqscan = off;

EXPLAIN SELECT * FROM users WHERE status = 'active';
-- Look for "Bitmap Index Scan" on idx_users_status
```

### Exercise 3: Stats and why they matter

```sql
-- View statistics for the users table
SELECT * FROM pg_stats WHERE tablename = 'users';

-- If a query is slow, the first step is often:
ANALYZE users;
```

## 🎤 Interview Question Practice

**Q1**: "Why might a query use a Seq Scan even if an index exists on the column?"

**Answer**: 
1. The table is very small (Seq Scan is cheaper than random index IO).
2. The query returns a high percentage of the table (e.g., >20%).
3. The column statistics are outdated.
4. The cost of random IO (`random_page_cost`) is set too high in configurations.

**Q2**: "What is the difference between `EXPLAIN` and `EXPLAIN ANALYZE`?"

**Answer**: `EXPLAIN` shows the execution plan generated by the optimizer based on estimated costs. `EXPLAIN ANALYZE` actually runs the query and reports real execution time, number of rows processed, and buffer usage. **Warning**: `EXPLAIN ANALYZE` will execute `INSERT/UPDATE/DELETE` statements, so wrap them in a transaction if you don't want to change data!
